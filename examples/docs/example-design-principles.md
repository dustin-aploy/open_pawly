# Example Design Principles

These examples are designed to be:

1. **Realistic** — they model bounded roles that a real organization might deploy.
2. **Pawprint-aligned** — they use the same Pawprint shape as `pawprint`.
3. **Role-bounded** — they emphasize what the worker must not do as much as what it may do.
4. **Operationally legible** — they include concrete handoff paths and clear boundaries.
5. **Demo-friendly** — they pair Pawprint worker cards with task docs and example output artifacts so adopters can see the end-to-end story.

This directory should help teams understand how Pawprint looks in practice without embedding runtime logic or validation logic here.
